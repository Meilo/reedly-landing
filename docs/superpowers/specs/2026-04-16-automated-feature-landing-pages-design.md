# Automated Feature Landing Pages вЂ” Design Spec

**Date:** 2026-04-16
**Status:** Approved
**Goal:** Automatically generate one SEO-optimized landing page per Reedly macro-feature to dramatically increase organic search visibility on Google and LLM-based search engines.

---

## 1. Context & Motivation

Reedly's landing site (reedly.ai) currently has ~28 indexed URLs: homepage, solution, pricing, comparison, blog articles, and legal pages. Feature-level content is limited to a few cards on the `/solution` page and blog articles.

Each of Reedly's macro-features (voice recording, AI report, territory syntheses, etc.) corresponds to real user search intents on Google. By creating a dedicated, SEO-structured landing page per feature, Reedly can capture search traffic from users looking for specific capabilities вЂ” dramatically expanding its organic footprint from ~28 to ~50 indexed pages.

## 3. Architecture

### 3.1 Approach: YAML Structured Data + Astro Template

Content is stored as structured YAML files. A single Astro template component renders all feature pages with the same design, animations, and CSS as the existing site. Content is generated by Claude API via a script, committed to git, and reviewed before deployment.

### 3.4 Feature Content YAML Schema

Each file in `content/features/[lang]/[slug].yaml` follows this structure:

```yaml
# SEO metadata
seo:
  title: string           # <title> tag, max 60 chars, includes "Reedly"
  description: string     # meta description, max 155 chars
  keywords: string[]      # 3-5 target keywords

# Hero section
hero:
  eyebrow: string         # short label (e.g., "Rapport IA")
  title: string           # H1, supports <br /> and <em> tags
  lead: string            # 1-2 sentence description
  cta_label: string       # CTA button text
  cta_url: string         # CTA link

# Problem section (3 cards)
problem:
  eyebrow: string
  title: string           # H2, supports <br /> and <em>
  lead: string
  cards:                  # exactly 3 items
    - title: string
      text: string
      icon: string        # icon identifier

# Solution section (3 steps)
solution:
  eyebrow: string
  title: string
  lead: string
  steps:                  # exactly 3 items
    - title: string
      text: string

# Benefits section (4-6 cards)
benefits:
  eyebrow: string
  title: string
  cards:                  # 4-6 items
    - title: string
      text: string
      icon: string

# Use Cases section (3-4 cards)
use_cases:
  eyebrow: string
  title: string
  cards:                  # 3-4 items
    - title: string
      text: string
      icon: string

# FAQ section (4-6 items, used for Schema.org FAQPage)
faq:                      # 4-6 items
  - question: string
    answer: string

# Internal linking
related_features:         # 2-4 related feature slugs
  - slug: string
    label: string
```

**Icon system:** The `icon` field references inline SVG icons. The existing site uses hand-written SVG paths in each component. The feature template will use a small icon map (15-20 icons covering common concepts: microphone, document, clock, users, brain, globe, building, flask, chart, download, shield, search, zap, target, layers). Claude's generated YAML picks from this predefined set. The icon map is defined once in a shared Astro component.

## 5. SEO Strategy

### 5.1 On-Page SEO

Each feature page includes:
- **Unique H1** containing the primary keyword
- **Meta title** (max 60 chars) with keyword + "Reedly"
- **Meta description** (max 155 chars) with keyword + value proposition
- **Canonical URL** pointing to itself
- **Hreflang** tags: `fr`, `en`, `x-default` (pointing to EN)
- **Open Graph** + **Twitter Card** meta tags
- **Breadcrumb**: Home > Solutions > [Feature Name]

### 5.2 Structured Data (Schema.org JSON-LD)

Each page includes:
- `BreadcrumbList` вЂ” for breadcrumb rich results
- `FAQPage` вЂ” for FAQ rich results (from the FAQ section)
- `SoftwareApplication` вЂ” linking back to the main Reedly app

### 5.3 Internal Linking (SEO Cluster)

## 6. Content Generation Script

### 6.1 Command Interface

```bash
# Generate content for a single feature (both FR and EN)
pnpm generate:feature <feature-id>

# Generate content for all features
pnpm generate:features --all

# Regenerate a specific language only
pnpm generate:feature <feature-id> --lang fr
```

### 6.2 Script Flow

1. Read `src/data/features.yaml` to find the target feature entry
2. Load product context:
   - Reedly product description (from monorepo CLAUDE.md)
   - Relevant domain entities (from `packages/domain/src/entities/`)
   - Relevant SQL migrations (for real field/feature knowledge)
   - Existing site content (for tone and vocabulary consistency)
3. Build a Claude API prompt with:
   - Product context
   - Feature description and target keywords
   - YAML schema to fill
   - SEO constraints (H1 keyword, FAQ answering "People Also Ask", content depth > 800 words)
   - Truthfulness constraint: never invent features that don't exist
   - Tone: professional but accessible, matching existing site copy
4. Call Claude API (`@anthropic-ai/sdk`, already a dependency)
5. Validate the returned YAML against the schema
6. Write FR and EN files to `content/features/[lang]/`
7. Update `public/sitemap.xml` with new entries
8. Print summary for human review

### 6.3 Prompt Strategy

The system prompt includes:
- Full Reedly product description
- The specific feature's database tables, domain entities, and capabilities
- The existing site's tone of voice (extracted from current copy)
- SEO best practices checklist
- The exact YAML schema to output
- Instruction to write in the target language (FR or EN) natively вЂ” not translate

Each language gets its own Claude call to ensure native-quality copy, not machine translation.

## 8. Bilingual Strategy

- Every feature has a FR and EN content file
- The `[...slug].astro` route in each language directory reads the corresponding language file
- Hreflang alternates link FR в†” EN for each feature pair
- Content is generated natively per language (not translated) for better SEO in each market
- The `features.yaml` registry maps FR slug в†” EN slug for hreflang generation

## 9. Out of Scope

- Feature pages for upcoming/unreleased features (future phase)
- Sector-specific variants (e.g., "AI Report for Tourism") вЂ” may be a future SEO expansion
- CMS or admin UI for managing features вЂ” YAML files in git are sufficient
- A/B testing of landing page variants
- Dynamic content (all pages are statically generated at build time)
